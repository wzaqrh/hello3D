寻路 - 导航网格寻路

寻路

游戏中我们经常要查找从一个位置通往另一个位置的路径。我们可以用图搜索算法来找到一条这样的路径。A*是一个比较出名的寻路算法。

参考链接：https://www.redblobgames.com/pathfinding/a-star/introduction.html

上面的链接比较详细的介绍了路径搜索算法。我们这里再对A*的算法做一些介绍。

A*

和其它的图搜索算法一样，A*潜在地搜索图中一个很大的区域。和Dijkstra一样，A*能用于搜索最短路径。和BFS（Breadth First Search）一样，A*能用启发式函数引导它自己。在简单的情况中，它和BFS一样快。所有基于图拓扑的结构都可以使用A*算法。

成功的秘决在于，它把Dijkstra算法（靠近初始点的结点）和BFS算法（靠近目标点的结点）的信息块结合起来。在讨论A的标准术语中，g(n)表示从初始结点到任意结点n的代价，h(n)表示从结点n到目标点的启发式评估代价。当从初始点向目标点移动时，A\权衡这两者。每次进行主循环时，它检查f(n)最小的结点n，其中f(n) = g(n) + h(n)。

如果不考虑具体实现代码，A*算法是相当简单的。有两个集合，OPEN集和CLOSED集。其中OPEN集保存待考查的结点。开始时，OPEN集只包含一个元素：初始结点。CLOSED集保存已考查过的结点。开始时，CLOSED集是空的。如果绘成图，OPEN集就是被访问区域的边境（frontier）而CLOSED集则是被访问区域的内部（interior）。每个结点同时保存其父结点的指针因此我们可以知道它是如何被找到的。

在主循环中重复地从OPEN集中取出最好的结点n（f值最小的结点）并检查之。如果n是目标结点，则我们的任务完成了，从目标节点反向遍历生成路径。否则，结点n被从OPEN集中删除并加入CLOSED集。然后检查它的邻居n’。计算到达n’的路径的代价g(n’)，设定为g(n) + movementcost(n, n’)。如果n’在OPEN集中，并且新计算的g值大于或等于原先存储的g值，那么它是以后肯定会被检查的，现在不需要考虑，但如果新计算的g值比原先存储的小就说明新的路径比原来的好，就要把这个邻居n'从OPEN集中删除，并重新添加到OPEN集中（可以通过用新的g值替换老的g值并用节点n带他原来的父节点的方式把两步合为一步）。如果邻居n’在CLOSED集中，那么它是已经被检查过的,如果它的新计算的g值更小，把这个邻居n’从CLOSED集中删除。否则，把它加入OPEN集，把它的父节点设为n。

下面用伪代码的形式实现A*算法。

首选是节点，前面说了，不只是二维的网格，所有是用图的方式组织的地图数据都可以用A*算法来实现寻路。所以这里介绍的节点是一个通用节点，不同的数据可以用具体的节点实现方法。

class AStarNode
    {
        //这个节点的父节点
        AStarNode Parent;
        ///到目前为止的累计消耗
        double GCost;
        ///到目标点的预估消耗
        double HGoalEstimate;
        /// 从起始点到当前点的消耗+从当前点到目标点的预估消耗
        double GetTotalCost();

        /// 目标节点.
        AStarNode GoalNode;
        /// 判断当前点是否是目标点
        bool IsGoal();

        // 估算从当前点到目标点的消耗
        double Calculate();
        // 获得和当前节点邻接的所有节点，获得节点时计算总消耗
        void GetSuccessors(ArrayList ASuccessors);
    }

接下来是实现OPEN集和CLOSED集所用到的类，这个数据结构有很多的实现方法，也是影响算法速度的关键因素之一。这里只写一个抽象的实现，具体的实现方法会在后面进一步讨论。

class DataHolder
{
    //返回集合中总消耗最小的节点并从集合中删除
    AStarNode Pop();
    //把一个节点加入到集合中
    void Push(AStarNode)；
    //检查集合中是否包含该节点
    bool Contains(AStarNode);
    //获得集合中的该节点
    AStarNode GetNode(AStarNode);
    //删除集合中指定节点
    void Remove(AStarNode);
    //集合中元素数量
    int Count;
}

最后是算法类,此类中把寻路的实现代码也加进来

class AStar
{
    //寻路的起点
    AStarNode StartNode;
    //寻路终点
    AStarNode GoalNode;
    //OPEN LIst
    DataHolder OpenList;
    //CLOSED LIST
    DataHolder ClosedList;
    //邻居节点
    ArrayList Successors;
    //最终路径
    ArrayList Path;

    //寻路实现
    void FindPath(AStarNode startNode, AStarNode goalNode)
    {
        StartNode = startNode;
        GoalNode = goalNode;

        OpenList.Push(StartNode); //把起点加入到OPENLIST中
        while(OpenList.Count > 0)
        {
            //获得OPENLIST集合中总消耗最小的节点
            AStarNode currentNode = OpenList.Pop();
            //如果这个节点就是目标点，反向查找生成路径
            if (currentNode.IsGoal())
            {
                while (currentNode != null)
                {
                    Path.Insert(0, currentNode);
                    currentNode = currentNode.Parent;
                }
                break;
            }
            //获得当前节点的所有邻居节点
            currentNode.GetSuccessors(Successors);
            foreach (AStarNode node in Successors)
            {
                //测试这个节点是否在OPEN LIST中，
                //如果在并且总的消耗值比新计算的节点的大,
                //就把节点从OPENLIST集合中删除
                //这里的实现方案会依据集合的实现方式不同而不同，这里默认集合内是已经排序好的，每次添加新元素都进行排序，所以这里的实现是先删除再添加，而没有直接改变已有元素的状态
                AStarNode nodeOpen = null;
                if (OpenList.Contains(node))
                {
                    nodeOpen = OpenList.GetNode(node);
                }    
                if (nodeOpen != null && node.GetTotalCost() > nodeOpen.GetTotalCost())
                    continue;
                //测试这个节点是否在CLOSED LIST中
                //如果在的话并且总的消耗比新计算的节点的总消耗大
                //就把节点从CLOSED LIST集合中删除
                AStarNode nodeClosed = null;
                if (ClosedList.Contains(node))
                {
                    nodeClosed = ClosedList.GetNode(node);
                }
                if (nodeClosed != null && node.GetTotalCost() > nodeClosed.GetTotalCost())
                    continues;
                //从OPENLIST中删除老的节点
                OpenList.Remove(nodeOpen);
                //从CLOSEDLIST中删除老的节点
                ClosedList.Remove(nodeClosed);
                //把新节点加入到OPEN LIST
                OpenList.Push(node);
            }
            ClosedList.Push(currentNode);
        }
    }
}

以下列出几个WayPoint的不足之处：

  1. 一些复杂的游戏地图需要的WayPoint数量过于庞大
  2. 有时会使角色走“Z”型路径

如下图A点和B点之间的路径

NAV寻路如下图

下图是路点寻路，如黄线，会产生“Z”字形

下图为文章开始时展示的地图的比较，红线是wayPoint寻路，兰线是nav。

3. 不能进行动态修正，如果地图中有动态障碍，处理起来会很困难

​ 如要实现即时战略游戏中，一辆在路上行走的坦克会挡住你军队的去路，这个移动的坦克就是一个动态障碍。

4. 不能根据角色的特性（不同宽度、高度等）改变路径

​ 如一个狭窄的通道，普通的人能够通过，而一辆马车的宽度超过通道宽度，应该不能通过。

5. 不能保存地形的附加信息，如地表高度、地面特征（水面、沙地等）

​ 比如一个游戏中的角色在走到沙地上时会降低移动速度，或走在一个斜坡上时人物会发生倾斜等。

Nav网格生成

假设上图是一个游戏地图，红色的区域是不可行走的区域，浅灰色区域是可行走区域，要想在游戏中实现nav寻路，必须将可行走区域转化为nav网格并保存为一种固定形式的数据，如下图浅红色的三角形。

nav网格必须是凸多边形，这里使用三角型，当然也可以使用四边形。使用Delaunay三角剖分算法可以生成Nav网格。

Nav网格寻路

上面已经介绍过：所有是用图的方式组织的地图数据都可以用A算法来实现寻路。只要把三维的网格结果组织成图的形式就能用A\算法。另一个就是消耗的算法。

绿色直线代表最终路径和方向，路径线进入三角形的边称为穿入边，路径线出去的边称为穿出边。每个三角形的花费采用穿入边和穿出边的中点的距离（图中红线）。H值为穿出边中点到寻路目标点的距离。这是一种常用的计算消耗的算法，当然也有其他改进算法。
